Adobe Commerce 2.4.6 End of Support — 63 Days to August 11, 2026

Adobe Commerce 2.4.6 reaches end of support on August 11, 2026. That is 63 days from today. After that date, Adobe will issue no further security patches, no quality fixes, and no official support for any issue you encounter. PCI compliance becomes a documented gap. Your codebase becomes an unpatched target.

If you are running 2.4.6 — or managing clients who are — you do not have time for a slow start. The question is not whether to migrate. The question is whether you know what you are actually migrating.

The Deadline Is Real and It Is Close

End of support does not mean your store stops working on August 12. Orders will still process. Checkout will still function. What changes is Adobe's obligation to you, which goes to zero. Every vulnerability discovered after August 11 in 2.4.6 core will remain permanently unpatched. Security researchers will document them. Exploit kits will be written against them. The CVE database will become a public map of undefended surface area in your production environment.

The pattern is documented. Magento 1 reached EOL in June 2020. Magecart attacks targeting known unpatched vulnerabilities swept through thousands of Magento 1 installations in the years that followed. The same dynamic applies to every unsupported version. Time on an unpatched platform is not neutral — it is accumulating risk.

PCI compliance note: PCI DSS Requirement 6.3.3 mandates that all system components are protected from known vulnerabilities by installing applicable security patches. Running on a platform with no available patches creates a documented compliance gap that QSAs are required to flag. For enterprise clients, this is not a theoretical concern — it is a real exposure in their next annual audit.

What Is Changing in the Migration

A 2.4.6 to 2.4.7 upgrade is not a version bump. The dependency stack changes substantially, and each change is a potential conflict point in your codebase.

// PHP 8.3 minimum, PHP 8.4 incoming

Adobe Commerce 2.4.7 requires PHP 8.3 at minimum. PHP 8.4 support is being added in patch releases. The practical consequence: any custom module or third-party extension using deprecated PHP 8.1 or 8.2 syntax — dynamic property creation, string interpolation patterns, certain reflection API usages — will generate deprecation notices or hard failures depending on your error configuration. This is not theoretical. It shows up in production.

The Ghost Architect conflict scanner identifies these PHP version compatibility issues as part of a standard scan. They surface as CRITICAL or HIGH findings with the specific files and line ranges where the incompatibilities live.

// OpenSearch replaces Elasticsearch

Adobe Commerce 2.4.7 drops Elasticsearch support in favor of OpenSearch 2.x. If your infrastructure is running Elasticsearch — which most 2.4.6 installations are — that is a platform-level change that needs to be planned for explicitly. Any custom search module, custom indexer, or extension that calls Elasticsearch APIs directly needs to be audited and updated before the migration goes live.

// MariaDB 11.4 and RabbitMQ 4.x

The supported database and message queue versions shift upward. MariaDB 11.4 introduces behavioral changes in certain query execution paths. RabbitMQ 4.x changes the management plugin API and deprecates several queue configuration patterns. Custom modules that interact with these systems at a low level — direct database queries, custom queue consumers — need to be tested against the new versions explicitly.

// GraphQL schema changes

Adobe Commerce 2.4.7 includes GraphQL schema changes that can break headless or PWA storefronts. Deprecated fields are removed. New required fields are added to certain query responses. Any custom GraphQL resolvers or schema extensions need to be validated against the updated schema before the migration goes live.

Why File-by-File AI Prompting Is Not Enough

The natural instinct when facing a migration like this is to open an AI assistant, paste in a file, and ask it to find compatibility issues. That works for individual files. It does not work for codebases.

The conflicts that actually break migrations are not confined to single files. They live in the relationships between files. A payment gateway that calls a custom cart model that overrides a core service that depends on a version-specific Elasticsearch API — that chain of dependencies does not appear when you review each file in isolation. You see the endpoint and you see the starting point, but you do not see the path between them.

That is the difference between file-by-file prompting and codebase-level analysis. One tells you what is in a file. The other tells you what breaks when you move.
